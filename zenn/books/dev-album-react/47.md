        # Partial Pre-rendering と最新レンダリング
        
          Next.js の Partial Pre-rendering（PPR）は、静的コンテンツと動的コンテンツを一つのページで最適に組み合わせる
          革新的なレンダリング手法です。従来の SSR / SSG / ISR を超える、次世代のレンダリングモデルを学びましょう。
        

        :::message
**なぜ今これを学ぶのか**
従来の Next.js では、ページ単位で「静的」か「動的」かを選ぶ必要がありました。
            PPR はこの制約を取り除き、ページの一部を静的に事前レンダリングしつつ、
            動的な部分だけをリクエスト時に生成します。これにより、初回表示速度と最新データの両立が可能になります。
:::

        
          
          
            ## PPR（Partial Pre-rendering）とは
            
              Partial Pre-rendering は、ビルド時に生成できる静的な「シェル」と、
              リクエスト時にサーバーで生成する動的な「コンテンツ」を一つのレスポンスとして返す仕組みです。
              ユーザーにはまず静的なシェルが即座に表示され、動的な部分はストリーミングで後から届きます。
            

            ### 従来のレンダリング方式との比較

            
              
                #### 従来の方式
                
                  
                    SSG

                    ビルド時に全て生成。高速だが動的データに対応できない

                  

                  
                    SSR

                    毎回サーバーで生成。最新データだが初回表示が遅い

                  

                  
                    ISR

                    定期的に再生成。静的と動的の折衷だがページ全体が対象

                  

                

              

              
                #### PPR
                
                  
                    &#9679;

                    ページの静的部分はビルド時に生成（CDN から配信）

                  

                  
                    &#9679;

                    動的部分だけリクエスト時にサーバーで生成

                  

                  
                    &#9679;

                    Suspense の境界で自動的に分離

                  

                  
                    &#9679;

                    SSG の速度と SSR の鮮度を両立

                  

                

              

            

            
              #### PPR の処理フロー
              
                
                  1

                  **ビルド時:** ページの静的部分（ナビゲーション、レイアウト、静的テキスト等）を HTML として事前生成。動的部分にはフォールバック（loading UI）を埋め込む

                

                
                  2

                  **リクエスト時:** 静的シェルを即座にレスポンスとして送信開始（CDN キャッシュから）

                

                
                  3

                  **ストリーミング:** 動的部分がサーバーで生成され次第、HTML ストリームとして追加送信。フォールバックが実際のコンテンツに置換される

                

              

            

            :::message
**PPR は実験的機能**
PPR は Next.js 15 時点ではまだ実験的（experimental）な機能です。
                本番利用は可能ですが、API が将来変更される可能性があります。
                next.config.ts で明示的に有効化する必要があります。
:::
          

          
          
            ## 静的シェル + 動的コンテンツ
            
              PPR の核心は、React の `Suspense` を使って
              静的な部分と動的な部分を自動的に分離することです。Suspense の境界の外側は静的シェルとしてビルド時に生成され、
              内側は動的コンテンツとしてリクエスト時に生成されます。

            :::message
**PPR の最大のメリット**
PPR では、開発者がページの分割方法を意識する必要がほとんどありません。
                通常通り Suspense を配置するだけで、
                Next.js が自動的に静的部分と動的部分を判別してくれます。
                これまで SSG と SSR のどちらを選ぶか悩んでいた場面で、PPR が最適解を提供してくれます。
:::
          

          
          
            ## PPR の設定と使い方
            
              PPR を使うには、`next.config.ts` で
              実験的機能として有効化し、ページ単位またはアプリ全体で適用を選択します。
            

            ```typescript
// next.config.ts
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  experimental: {
    // PPR を有効化（incremental モード）
    ppr: 'incremental',
  },
};

export default nextConfig;

// 'incremental' を指定すると、ページ単位で PPR を有効にできる
// true を指定するとアプリ全体で PPR が有効になる
```

            :::message alert
**動的関数に注意**
PPR では、cookies()、
                headers()、
                searchParams などの
                動的関数を Suspense の外側で使うと、ページ全体が動的レンダリングになり PPR の効果が失われます。
                動的関数は必ず Suspense の内側のコンポーネントで使用してください。
:::

            ## dynamicIO
            
              `dynamicIO` は Next.js 15 で導入された実験的な機能で、
              データ取得の動的・静的判別をより厳密に制御します。有効にすると、キャッシュされていないデータアクセスが
              Suspense の外側にある場合にビルドエラーが発生し、PPR の最適な設計を強制します。
            

            ```typescript
// next.config.ts
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  experimental: {
    ppr: 'incremental',
    dynamicIO: true, // dynamicIO を有効化
  },
};

export default nextConfig;
```

            :::message
**'use cache' ディレクティブ**
dynamicIO と共に導入された
                'use cache' ディレクティブは、
                コンポーネントや関数のレンダリング結果をキャッシュするための仕組みです。
                cacheLife() でキャッシュの有効期間を、
                cacheTag() でタグベースの無効化を制御できます。
                これらは 'use server' や
                'use client' と同じディレクティブの仕組みです。
:::

            ```tsx
// 'use cache' の実践的な使い方

import { cacheLife, cacheTag } from 'next/cache';

// コンポーネント単位のキャッシュ
async function ProductList() {
  'use cache';
  cacheLife('minutes');         // 数分間キャッシュ
  cacheTag('product-list');     // タグで管理

  const products = await db.product.findMany();

  return (
    
      
    

  );
}

// 関数単位のキャッシュ
async function getCategories() {
  'use cache';
  cacheLife('days');             // 日単位でキャッシュ
  cacheTag('categories');

  return await db.category.findMany();
}

// Server Action でキャッシュを無効化
'use server';
import { revalidateTag } from 'next/cache';

export async function addProduct(formData: FormData) {
  await db.product.create({ ... });
  revalidateTag('product-list'); // キャッシュを無効化
}
```

            ## レンダリング方式の進化まとめ
            
              
                
                  SSG（静的生成）

                  ビルド時に全ページを HTML 化。ブログや LP に最適。データ更新には再ビルドが必要

                

                
                  SSR（サーバー生成）

                  リクエストごとにサーバーで HTML 生成。常に最新データだが、レスポンスが遅くなりがち

                

                
                  ISR（増分静的再生成）

                  定期的に静的ページを再生成。SSG と SSR の中間だが、ページ全体が対象

                

                
                  PPR（部分事前生成）

                  ページ内の静的部分と動的部分を自動分離。Suspense ベースでベストな組み合わせを実現

                

                
                  dynamicIO + use cache

                  コンポーネント・関数単位の細粒度キャッシュ制御。PPR と組み合わせて最適なパフォーマンスを実現
